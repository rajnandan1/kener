# Confirmation Threshold damps the recorded timeline at write time, with retroactive backfill

A per-monitor `confirmation_threshold` (integer, default 1 = off, capped at 60) makes the recorded status timeline ignore transient flapping: the monitor's recorded side ‚Äî healthy (`UP`) versus unhealthy (`DOWN`/`DEGRADED`) ‚Äî only flips after that many consecutive scheduled-check observations land on the other side. Shorter runs never surface as a status change. This is implemented in the scheduled-check write path (`monitorExecuteQueue`), so a new `monitoring_data.raw_status` column records what each check actually observed while `status` holds the confirmed side. When a flip is confirmed, the pending window is backfilled: the observations leading up to it are rewritten to their observed side, so a confirmed outage reads as down from its first failing check, not from the moment of confirmation. The same rule runs symmetrically on recovery. The net invariant is simple to state: **the recorded timeline equals the raw observation timeline with every run shorter than the threshold collapsed into its surrounding side**; the threshold only delays the *live* display by N samples and erases sub-threshold blips.

Issue #712 drove this: ping/API monitors that drop a single check (a public resolver timing out, an HTTP blip) get recorded as `DOWN`, which dents uptime % and the day bars even though the service never actually went down, and the only remedy was manually rewriting history through the UI. The existing alerting Failure/Success Thresholds (`consecutivelyStatusFor` et al.) only gate *notifications and incidents* ‚Äî the recorded timeline, day bars, and uptime % still showed every blip, which is exactly the gap reported.

Damping happens at **write time**, not read time. The rejected alternative ‚Äî record raw truth and apply the threshold in every consumer (uptime calculator, day-bar aggregation, current-status, badges, embeds, group scoring) ‚Äî keeps the data pristine but forces the same rule into N readers, where any one that forgets it leaks the blip; it is a permanent tax for a property that is cheaply enforced once at the single ingestion choke point. Writing the confirmed side once means uptime %, the bars, badges, alert evaluation, and group scoring all inherit the damped truth for free. The cost accepted: `status` is no longer literally "what the check returned" ‚Äî it is "the confirmed side" ‚Äî which is precisely the semantic this feature defines; `raw_status` keeps the original observation recoverable and serves as a forensic trail of absorbed blips.

The flip is **retroactive**, not forward-only. Forward-only (first N‚àí1 failing samples stay `UP` forever, `DOWN` begins at confirmation) is simpler ‚Äî history would stay append-only ‚Äî but it systematically shaves N‚àí1 minutes off the front of every real outage, permanently flattering uptime and misreporting outage start times. Retroactive backfill rewrites a bounded window (at most threshold‚àí1 rows, one `UPDATE`) at the moment of crossing; it also makes the threshold compose with the alert Failure Threshold by `max` rather than `+`, because once backfilled the alert lookback sees N consecutive confirmed rows immediately. The cost accepted: `monitoring_data` is no longer append-only over the pending window, so the last-value cache and any reader that observed those rows mid-window saw the pre-flip side until convergence.

Deliberately accepted boundaries, chosen for one uniform rule rather than special cases:

- **Count, not minutes.** The threshold is consecutive observations, well-defined for any cron, matching the alerting thresholds' unit; for the common every-minute cron, count and minutes coincide.
- **One symmetric threshold.** A single binary UP‚Üînot-UP flip is damped in both directions; severity changes *within* an already-unhealthy stretch (`DOWN`‚Üî`DEGRADED`) commit immediately, since the side is not changing. A flapping `DOWN`/`DEGRADED` service therefore still confirms unhealthy (both are not-UP), and a lucky single success mid-outage no longer paints a false-green minute. Two independent thresholds (separate confirm/recover) were rejected as config surface without a public-timeline use case ‚Äî operational flexibility already lives in the alert configs.
- **Overlays bypass and freeze.** Incident/maintenance overlays are admin-asserted truth: they win on `status` immediately, and the confirmation count freezes while one is active (the lookback skips overlay rows), mirroring the alert-window freeze in ADR 0005. Maintenance is expected downtime, so normal monitoring resumes with a fresh count.
- **Scheduled checks only.** Only `REALTIME`/`TIMEOUT`/`ERROR` observations count. `MANUAL` data-API pushes (a separate write path, explicit assertions), `DEFAULT_STATUS` fill, and `NO_DATA` are transparent to the counter; `NO_DATA` is written honestly as grey and never absorbed by a backfill.
- **Uniform across types, opt-in, default off.** Every monitor type runs through the same path with no per-type branching ‚Äî a no-op where nothing flaps, stacking harmlessly with Heartbeat's own time thresholds and with member-level grace on Group Monitors. A brand-new monitor's first observation bootstraps the side immediately (no fictional `UP`).
- **Backfill does not cascade to groups.** A Group Monitor scores on members' *confirmed* current status, so member grace propagates into groups for free; but a member's retroactive backfill does not rewrite the history of groups it belongs to, leaving a bounded (< threshold) lag between a member's backfilled timeline and a parent group's recorded one. Cross-monitor retroactive correction was rejected as a materially larger feature with unbounded cascade.
